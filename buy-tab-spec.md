# easyinvesting.app 「매수」 탭 — 구현 명세서 (리버스 엔지니어링)

> 검증 기준: 2026-07-21, https://easyinvesting.app/#/buy 실제 API 응답·UI·JS 번들 문자열을 직접 확인하여 작성.
> 표기: ✅ = 실데이터로 검증된 사실, 🔶 = 데이터 정황상 강한 추정, ❓ = 서버 내부 로직이라 외부에서 관측 불가(구현 시 자체 결정 필요).

## 1. 개요

매수 탭은 **"지금 살 만한 종목을 매수 근거별로 나누고, '과거에 바로 이 자리에서 샀다면 어땠는가'를 백테스트 성적표로 증명해서 보여주는"** 화면이다. (사이트 자체 설명 문구 확인 ✅)

두 개의 하위 탭으로 구성되며, 서로 완전히 다른 로직·데이터·갱신주기를 가진다.

| 하위 탭 | 근거 | API | 갱신 주기 |
|---|---|---|---|
| 추세추종 | 기술적 신호(이평 눌림·박스 돌파·신고가) + 상대강도 | `GET /api/buy-signals?limit=1000` | 일 1회 (관측: `built_at 2026-07-20T22:30:28Z`) ✅ |
| 계절성 | 달력 기반 계절성 패턴 | `GET /api/buy-signals/seasonal?limit=2000` | 월 1회 (관측: `built_at 2026-07-01`, 배너 "월간 갱신") ✅ |

**아키텍처**: 서버(배치)가 모든 계산을 미리 끝내 놓고, 프론트는 완성된 목록 JSON을 받아 필터·정렬·표시만 한다. 카드 펼침 시에만 상세 API를 추가 호출한다. ✅

## 2. 추세추종 탭

`GET /api/buy-signals?limit=1000` → `{ meta, count, total, page, limit, sort, rows }`

### 2.2 신호(매수 근거) 3종 정의 ✅

| signal | detail 형식 | 관측된 detail 값 | 의미 |
|---|---|---|---|
| 이평 (눌림목) | `{L\|S}/{SMA\|EMA}{기간}` | SMA·EMA × {10,20/21,30,40,50,60} × L/S = 24조합 | 상승 추세 종목이 이동평균선까지 눌렸을 때. L=장기, S=단기 추세 맥락 |
| 박스 (박스권 돌파) | `L` 또는 `S` | L, S | 박스권 상단 상향 돌파. 장기/단기 박스 |
| 신고가 | `ATH` \| `20` \| `55` | ATH, 20, 55 | 역대최고가 / 20일 신고가 / 55일 신고가 돌파 |

- 이평선 세트: SMA 10·20·30·40·50·60 + EMA 10·**21**·30·40·50·60 (EMA만 20 대신 21 사용) ✅
- phrase 생성 규칙 ✅: 이평 → `"{MA종류}{기간} {장기|단기} 근접 · {N}일 보유"`, 박스 → `"박스돌파 {장기|단기} · {N}일 보유"`, 신고가 → `"{역대최고|20일신고가|55일신고가} 돌파 · {N}일 보유"`
- "터치/근접" 판정 임계값(이평선에서 몇 % 이내면 신호인지), 박스 정의 파라미터(기간·진폭·유지기간)는 ❓ (참고: 알람 기능에는 "박스 진폭", "최소 유지 기간" 입력이 존재 — 박스 정의에 같은 개념 사용 추정 🔶)

### 2.3 후보 자격(1차 필터) 

- **종합 RS ≥ 90** (`rs_min: 90`) ✅ — 시장 대비 상대강도 상위 10%만. rows의 rs 실측 범위도 90 이상.
- **표본수 n ≥ 20** (`min_sample: 20`) ✅ — 과거 동일 신호 발생이 20회 미만이면 제외. (사이트 다른 화면 문구: "표본 5 미만 = 신뢰도 낮음")
- **현재가가 매수 구간 안에 있는 종목만 노출** — 270/270 전 행이 `zone_low ≤ cur_price ≤ zone_high` ✅. (구간을 벗어나면 목록에서 빠지는 것으로 추정 🔶)
- RS 데이터가 오래되면(예: `rs_kr: false`) 해당 국가 신호가 0건이 되고 배너로 경고. ✅

### 2.4 백테스트와 최적 보유기간 선정

각 (종목 × 신호 조합)에 대해 **8개 보유기간**(이평·신고가 {2,3,5,10,20,60,126,252}일, 박스 {2,3,5,10,20,63,126,252}일)의 과거 성적을 계산한다(카드 펼침의 "8기간 성적표" ✅). 각 기간별 산출 지표: `win_rate, avg_win, avg_loss, pl_ratio, 표본수`.

**목록에 표시되는 대표 성적 = 8기간 중 최적 보유기간 1개의 성적**이며, 최적 기간 선정 규칙은 다음 두 가지로 검증됨:

1. **표본수 ≥ 20(min_sample)인 기간만 후보** — 실례(ESTA·신고가 ATH): 5일 보유가 EV 0.77로 3일(EV 0.71)보다 높지만 표본 15로 탈락, 표본 20인 3일이 선택됨 ✅
2. 후보 중 **EV(R배수 기대값) 최대** 기간 선택:

```
EV = ( p × avg_win + (1 − p) × avg_loss ) / |avg_loss|,  p = win_rate/100
```

- 270행 중 268행이 이 공식과 오차 0.02 이내로 일치 ✅ (2행 불일치는 공개 필드의 반올림 때문 — 서버는 미반올림 값으로 계산 🔶)
- `pl_ratio = avg_win / |avg_loss|` 266/270 일치 ✅ (동일한 반올림 이슈)
- **손실 표본이 없는 기간(avg_loss 없음)은 EV를 정의할 수 없어 선택에서 제외됨** — 실례: RSI의 252일 보유는 승률 100%(표본 20)인데도 126일이 선택됨. 성적표에서 해당 행의 평균손실·손익비가 "—"로 표시 ✅
- 목록 row의 `win_rate/avg_win/avg_loss/pl_ratio/n`은 선택된 기간의 성적표 행과 정확히 일치 ✅ (RSI 126일 행, TRGP 252일 행으로 대조 확인)

### 2.5 매수 구간(zone) 산출 공식 ✅ (전 행 실측으로 도출)

| signal | zone_low | zone_high | 실측 근거 |
|---|---|---|---|
| 이평 | `ref_price × 0.98` | `ref_price × 1.03` | low/ref = 0.978~0.983, high/ref = 1.029~1.034 (센트 반올림 오차 범위) |
| 박스 | `ref_price` (박스 상단) | `ref_price × 1.05` | ref = zone_low 정확 일치, 폭 4.95~5.08% |
| 신고가 | `ref_price` (돌파선) | `ref_price × 1.05` | 동일 |

즉, 이평은 "이평선 −2% ~ +3%", 돌파형은 "돌파선 ~ +5%"가 매수 유효 구간.

### 2.6 별점(★) 로직 — 감점 태그 방식 ✅ (270행 전수 분석으로 도출)

주식에만 부여(ETF는 null). **기본 3★에서 감점 태그(cut_reason)에 따라 강등**되는 구조가 데이터와 정확히 일치:

| stars | 조건 (실측) | 분포 |
|---|---|---|
| ★★★ | 감점 태그 0개 (cut_reason = null) | 113행 전부 태그 0개 |
| ★★ | 경미 태그 1~3개 | 태그 1개=52, 2개=37, 3개=40 |
| ★ | **"대형주 무리한급등"** 태그 보유 | 13행 전부 이 태그 |

관측된 감점 태그 4종 (복수면 `" / "`로 결합) ✅:

1. `변동성 최상위(Q4)` — 변동성이 유니버스 상위 25%
2. `윗꼬리 5일+` — 윗꼬리 캔들 5일 이상
3. `RS최고 97+(초과열)` — RS가 97 이상(과열)
4. `대형주 무리한급등(시총대형 & 직전20일≥+50% | 60일최대일간≥+20%)` — 즉시 ★1

meta의 부가 개념: `컷`(완전 제외, 현재 0), `lottery`(복권성, 현재 0), `star_cut: {KR: 4}`(KR 전용 컷 기준 ❓), `ax3_unmeasured`/`ax6_nodata`(내부 평가축 3·6번의 측정불가 카운트 — 축 정의는 ❓, 별점 결정에는 위 감점 규칙만으로 현재 데이터가 100% 설명됨).

각 태그의 정확한 산식(변동성 Q4 기준, 윗꼬리 판정식)은 ❓ — 태그 이름에 파라미터가 일부 명시됨(직전 20일 ≥ +50%, 60일 내 최대 일간 ≥ +20%).

### 2.7 프론트 UI 명세 ✅

**상단 배너**: `신호 기준일 {built_at의 날짜}` + 데이터 지연 경고(예: "KR RS 지연 — 신호가 묵었을 수 있음"). 지연이 있으면 붉은 톤.

**필터 칩 (클라이언트 사이드 필터링)**:
- 국가: 전체 / 한국 / 미국
- 자산: 전체 / 주식 / ETF
- 신호: 전체 / 이평 / 박스 / 신고가
- 시총: 전체 / ≥$1B / ≥$10B / ≥$100B / ≥$1T

**정렬 셀렉트**: `별 우선`(기본, value=stars) / `EV순` / `승률순` / `RS순`
- 별 우선 = `stars` 내림차순(null인 ETF는 최후미) → 동률 시 `ev` 내림차순. 이 규칙으로 UI 상위 8위 순서가 정확히 재현됨 ✅

**표시 개수**: 필터·정렬 후 상위 **100종목**까지만 렌더("100종목" 카운터 라벨) ✅

**카드(접힘)**: 순번, ★~★★★, 종목명, 신호 배지(이평=파랑, 박스=초록, 신고가=주황 계열), `phrase`, 한 줄 문구 "과거 데이터상 꾸준한 유형입니다", 펼침 화살표. 2열 그리드.

**카드(펼침)** — 이 시점에 상세 API 추가 호출:
- 섹터·산업 RS 칩: `Consumer Cyclical 45` `Gambling 37` — 값은 `GET /api/rs/us/sectors`, `GET /api/rs/us/industries`의 그룹 평균 RS ✅ (sectors 응답 필드: `sector, avg_rs, avg_1m, avg_3m, avg_6m, avg_12m, count, pct_90_overall, pct_90_1m`)

  - 이평 → `GET /api/ma-pullback/by-ticker?ticker={T}` (표본 컬럼명 "터치수")

## 3. 계절성 탭

### 3.1 API 응답 구조

`GET /api/buy-signals/seasonal?limit=2000` → `{ meta, count, total, page, limit, rows }`

#### meta (✅ 실측)

```json
{
  "built_at": "2026-07-01T00:58:14.148565",
  "today_doy": 202,
  "window": [200, 207],
  "total_candidates": 10,
  "cadence": "monthly",
  "win_cut": 90,
  "periods": [10, 15, 20]
}
```

**선정 로직** (배너 문구 "월간 갱신 · 오늘 기준 매수창(-2~+5일) · 승률>90%" 와 meta가 일치 ✅):

1. 월 1회 배치로 전 유니버스의 계절성 패턴(매년 특정 날짜 매수 → 특정 날짜 매도)을 백테스트.
2. **매수 시점이 오늘(DOY 202) 기준 −2일 ~ +5일 창(window [200, 207])에 들어오는 패턴만** 노출.
3. **승률 > 90%** (`win_cut: 90`)인 패턴만.
4. 검증 기간은 **10 / 15 / 20년** 중 하나(`period_years`), 표본수 = 연수(연 1회 발생이므로).

#### rows[i] 필드 (✅ 실측)

| 필드 | 예시 | 의미 |
|---|---|---|
| `ticker` | "PRG", "DOM.WA", "JPYKRW=X" | 티커(해외 거래소 접미사, 환율 포함) |

| `period_years` | 10 | 검증 기간(년) |

## 4. 이걸 그대로 구현하려면 — 시스템 구성

### 4.1 백엔드 (배치 2본 + 정적 API 5본)

**일간 배치 (추세추종)** — 장 마감 후 실행:
1. 가격 이력 수집(수정주가) — 미국/한국 주식 + ETF.
2. 종목별 RS 점수 계산(0~100 백분위). 사이트 설명상 "종합 RS = 1·3·6·12개월 시장 대비 초과수익 종합" 🔶, 정확 가중치는 ❓ → 자체 정의 필요 (통상 IBD식: 3·6·9·12개월 수익률 가중 백분위).
3. RS ≥ 90 종목에 대해 신호 검출: 이평 근접(24조합), 박스 돌파(L/S), 신고가(ATH/20/55).
4. 신호별 과거 전 구간 백테스트: 신호 발생일 매수 → 8개 보유기간(§2.4) → 기간별 win/avg_win/avg_loss/pl_ratio/표본수.
5. 티커·신호별 최적 기간 = 표본 ≥ 20인 기간 중 EV(R배수) 최대(무손실 기간 제외).
6. 티커당 대표 신호 1개 선정(EV 최대 🔶), zone 계산(§2.5), 현재가 zone 내 필터.
7. 별점: 감점 태그 산출(§2.6) → 3/2/1★.
8. 결과를 JSON으로 저장 → `/api/buy-signals` 서빙.

**월간 배치 (계절성)** — 월초 실행:
1. 글로벌 유니버스(주식+환율)에 대해, 후보 (매수일, 매도일) 조합의 계절성 백테스트를 10/15/20년 각각 수행. 후보 조합 생성 방식(모든 날짜쌍인지, 보유일 {5,15,25,30…}일 그리드인지)은 ❓ — 관측된 hold_days는 5/15/25/30일.
2. 승률 > 90% 패턴만 채택.
3. 서빙 시(또는 배치 시) 오늘 DOY 기준 [−2, +5] 창에 entry가 걸리는 것만 반환.

**정적 API 7본**: `/api/buy-signals`, `/api/buy-signals/seasonal`, `/api/ma-pullback/by-ticker`, `/api/ma-pullback/box-by-ticker`, `/api/nhigh/by-ticker`, `/api/rs/{us|kr}/sectors`, `/api/rs/{us|kr}/industries`

### 4.2 프론트 로직 (전부 본 문서로 재현 가능)

- 목록 1회 fetch → 클라이언트에서 필터 4종 + 정렬 4종 + 상위 100개 렌더.
- 카드 펼침 시 상세 API 3본 lazy fetch.
- 시총 필터: `market_cap_usd`가 null이면 "전체"에서만 노출 🔶.

### 4.3 필요한 원천 데이터

일별 OHLCV(수정주가) 10~20년치(계절성은 20년), 시가총액, 섹터/산업 매핑(yfinance 계열 명칭 관측: "Consumer Cyclical", "Credit Services" 등 🔶), ETF 카테고리("Japan Stock" 등 Morningstar 계열 명칭 🔶), 벤치마크 지수(^GSPC, ^KS11), 환율 시세(계절성용).

## 5. 그대로 구현 가능한가 — 갭 분석

**문서만으로 즉시 재현 가능한 것 (✅ 층)**
- 화면 구조, 필터·정렬·표시 규칙 전부
- API 스키마와 응답 형식 전부
- EV 공식, 손익비 공식, 최적 보유기간 선정 규칙
- 매수 구간(zone) 공식 3종
- 별점 3단계 규칙과 감점 태그 4종의 존재·결합 방식
- 계절성 선정 파이프라인(매수창 ±일수, 승률컷, 검증연수, 정렬)
- phrase 등 표시 문자열 생성 규칙

**자체 결정이 필요한 것 (❓ 층) — 결과 숫자가 원본과 달라질 수 있는 지점**
1. **RS 산식**: 기간별 가중치·백분위 산출 방법. (원본 재현이 목표가 아니라면 IBD식 가중 백분위로 대체 가능)
2. **이평 "근접" 임계값**: 종가가 이평선에서 몇 % 이내일 때 신호로 보는지. (zone이 −2%~+3%이므로 근접 판정도 유사 범위로 추정 🔶)
3. **박스 정의**: 박스 성립 조건(기간·진폭)과 돌파 판정. (알람 기능의 "박스 진폭·최소 유지 기간" 개념 재사용 추정 🔶)
4. **백테스트 진입 규칙 세부**: "터치" 1회의 정의(연속 근접일 중복 제거 방식 등).
5. **감점 태그 산식 세부**: 변동성 Q4의 변동성 정의, 윗꼬리 판정식. (대형주 무리한급등은 태그 문구에 파라미터 명시됨: 시총 대형 & 직전 20일 ≥ +50% 또는 60일 내 최대 일간 ≥ +20%)
6. **market_s(시장 레짐)** 판정식, **star_cut·lottery·ax3/ax6** 세부.
7. **계절성 후보 (entry, exit) 조합 생성 방식**.

**결론**: 화면·API·핵심 수식(EV, zone, 별점 매핑, 필터 기준치)은 완전히 해독되어 그대로 구현 가능하다. 다만 위 7개 항목은 서버 내부 파라미터라 관측이 불가능하므로, 자체 기준을 정해야 하며 그 경우 **구조와 UX는 동일하되 개별 종목 선정 결과는 원본과 일부 달라질 수 있다.** 원본과 결과까지 일치시키려면 해당 파라미터를 실험적으로 역산(예: 여러 날짜의 목록 스냅샷과 가격 데이터 대조)하는 추가 작업이 필요하다.
